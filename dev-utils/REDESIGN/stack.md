# Website Redesign Plan

The website will undergo a complete redesign to modernize its look and feel, making it much more professional and expandable.

## Website Info

The website is self-hosted on a mac, no VPS. SSD storage. Cloudflare is used. 

Traffic is low, only a few hundred unique visitors per day, that is our scale.

## Pre-Redesign Infrastructure Work

- **Live game state persistence to SQLite.** On each move, write game state (game JSON, clock values, whose turn, timestamp) to a `live_games` table. On server startup, rehydrate in-memory game objects from this table. This enables the server to resume interrupted games after a restart instead of losing them. Do not yet worry about adding time to their clock depending on how long the server was down.

- **Process manager for the production server process.** The Node.js server must run independently of any terminal or IDE session, survive reboots, and support zero-downtime reloads. Use PM2. Zero changes to directory structure — server still runs from the same working directory. PM2 only manages the running process itself. PM2 installs once on the machine, not as a project dependancy. Manual update deploy becomes: `git pull && npm ci --silent && npm run build && pm2 reload infinitechess` in any terminal window that is cd'd into the project directory. The terminal may be safely closed afterwards. To keep track of whenever the server restarts, add a new `startupLog.txt` log and log the start date and time to this log on server startup, e.g. `2026-03-10 14:32:03 | Server started. PID: 5389`. To view server console logs while running under PM2, use `pm2 logs infinitechess`. To monitor the amount of space logs take up, use `du -sh ~/.pm2/logs/*`.

- **Automated deployment via self-hosted GitHub Actions runner.** Install the runner as a system service on the server machine (starts automatically on boot, requires no ongoing maintenance). All development and PR merges go into `main`. When ready to release, merge `main` → `prod`. The deploy workflow has three triggers: (1) push to `prod` — full deploy (`git pull && npm ci && npm run build && pm2 reload infinitechess`); (2) `workflow_dispatch` — manual trigger from the GitHub Actions UI or via `gh workflow run deploy.yml`, with an optional `warning_seconds` input (GitHub UI renders this as a form field with the default pre-filled); (3) `repository_dispatch: hydrochess-release` — triggered from hydrochess's `build-wasm.yml` after a new engine release is published, which rebuilds (downloading the new WASM artifacts), and reloads (skipping `git pull`). The runner interacts with the live server by making an HTTP request to a new endpoint (`POST /API/prepare-restart`) — since the runner runs on the same machine, it hits `localhost`. That endpoint broadcasts a countdown warning to all connected clients. The `allowinvites.json` file and its polling mechanism are removed entirely — game state persistence makes blocking new games unnecessary. When the countdown expires, clients currently in a game see "Server is restarting, your game will resume soon" rather than "Server is restarting shortly". After the warning period, the runner proceeds with the deploy sequence and calls `pm2 reload`. Generate a RESTART_SECRET, put it in the web server's .env, and in the GitHub Actions secrets. Then the runner will include this secret in the request header when calling the endpoint.


## Redesign Stack and Principles

- Proper MPA. Professional. All features of the game aren't crammed into one giant page.
- Bandwidth-aware. Doesn't send all scripts across the whole page for one page. Break up script dependancies so each page doesn't request more js code than it needs to run its own page. Perhaps this matters a little bit less since all scripts are indefinitely cached.
- When requesting a new page, use SSR, server renders the entire html before sending it to the user. Client does not have to modify the header itself depending on session info, or fetch notification count. Server renders depending on authentication of the user. SSR anything that affects the visible first paint - header, notification badge count, member profile data, news "NEW" badges, etc. Stick to client-fetch for items trigerred by user interaction or things that update live - leaderboard "Show More" (use snabbdom for this), editor saves, preferences writes, etc.
- Use snabbdom to update page contents for in-page reactivity (updating a leaderboard list, a chat window, a game panel in real-time without full re-renders). Don't use it for things like rotating the fairy piece carasel in the Guide, because that is static and known at author time. Use Snabbdom when DOM content is **generated from data** - data-driven lists.
- Localization/translation system compatible with Weblate. One component (json/toml) per website feature (e.g. header nav, game UI, settings panel, leaderboard, profile page — each its own file). Dozens of components. Versioning support. Fallback to english if language-specific strings are missing. Markdown and article-like files don't use Weblate but may be translated manually by translators. Component's no longer need a version field for versioning, Weblate handles stale strings automatically when an English string changes, and marks all other languages' strings as stale until translators update them. So we no longer need removeOutdated(), we accept stale translations and render them instead of falling back to english for outdated strings. We do keep deepMerge() which is still needed for strings that are entirely missing (fallback to english).
- No translations for ToS/Privacy Policy. Only in english. Sourced from markdown files. Include a notice that on conflicts between the english and translated terms, the english version takes control. With markdown files, it's too hard to version control, telling translators what changed, when something changes, so we avoid translating them.
- CSS custom property theme system. A `[data-theme]` attribute on `<html>` (e.g. `data-theme="dark"`) selects a block of ~30 semantic CSS variables (e.g. `--c-bg`, `--c-surface`, `--c-text`, `--c-brand`, `--c-border`). Theme variables live in the shared stylesheet. Switching theme is one `setAttribute` call + `localStorage` write. A small inline script in `<head>` reads `localStorage` before any CSS loads to prevent flash of wrong theme.
- Content-hashed asset filenames. Build pipeline emits `styles.[hash].css` and `main.[hash].js`. These files are served with `Cache-Control: immutable, max-age=31536000` so browsers never re-request them. When a file changes, its hash changes and the new URL is fetched fresh. HTML is served with `Cache-Control: no-store` and embeds the correct hashed filenames. This gives both instant cache hits and instant updates with no coordination overhead. Images and other static assets referenced directly in Nunjucks templates (e.g. `<img src="/img/king_w.png">`) are not auto-hashed by the build pipeline — serve these with `Cache-Control: max-age=31536000` (without `immutable`) and manually add a `?v=2` query string in the template if the file ever changes. Reserve `immutable` only for build-pipeline-hashed files.
- Build tool: extend the existing esbuild pipeline in `build/`. Two additions required to `build/client.ts`: (1) `entryNames: '[dir]/[name]-[hash]'` to enable content-hashed output filenames; (2) `metafile: true` plus a small `writeManifest()` post-build function that reads esbuild's input→output map and writes `dist/manifest.json`. The server loads this manifest once at startup and passes hashed filenames as variables into every Nunjucks render. `build/views.ts` is deleted entirely — it only existed because the old server had no SSR capability, so every EJS template had to be pre-rendered × every language and written to `dist/client/views/` as static `.html` files for `res.sendFile()`. With Nunjucks SSR, HTML is rendered at request time so nothing is pre-built to disk. `dist/client/views/` no longer exists. `root.ts` switches from `res.sendFile()` to `res.render()`, and `nodemon.json` no longer needs to watch `src/client/views`.
- Header layout without JS measurement. Server-side auth renders the correct header state using CSS container queries or a CSS-only overflow fallback. Language-width variation is the remaining challenge.
- `<link rel="modulepreload">` for page entry points. Injected into each page's HTML so the browser can fetch all JS modules in parallel from the first response, eliminating import waterfall round trips. Do this very last when the imports for each page are finalized.
- Nunjucks as the server-side templating engine, replacing EJS. Key benefit: layout inheritance — one `layout.njk` defines the full `<html>`/`<head>`/`<body>` shell with named `{% block %}` slots; every page file just `{% extends "layout.njk" %}` and fills in its title, page-specific styles, and content. Adding a favicon or global meta tag means editing one file instead of every page. Logic stays in route handlers; templates only use `{% for %}` / `{% if %}` for iteration and conditionals.
- CSS methodology: One stylesheet for storing shared styles across all pages, plus individual stylesheets for each page. Short descriptive class names scoped via native CSS nesting — no BEM, no prefixes. Every page's CSS has one top-level wrapper matching the page's `<main>` class (e.g. `class="login"` on `<main>`, all that page's rules nested inside `.login {}`), so `.form-field` inside `.login` never bleeds elsewhere. lightningcss in the existing build pipeline transpiles nesting for older browsers automatically. Utility classes (`.hidden`, `.italic`, `.flex`, etc.) are hand-rolled and added to the shared stylesheet as-needed when redundancy is noticed — no Tailwind. CSS is colocated with the component it styles (e.g. `src/client/components/header/header.css`);
- Auth mechanism: keep the existing dual-token system unchanged — no backend migration needed. The refresh token is already an `httpOnly` cookie (`jwt`) sent automatically by the browser on every request including page navigations. `verifyJWT` middleware already reads it and sets `req.memberInfo` on every request, so Nunjucks SSR can render auth-aware HTML (correct header state, notification count, etc.) with zero additional work. Access tokens (short-lived, stored in memory by `validatorama.ts`) are retained for API calls — they skip the DB lookup when fresh and are fully encapsulated behind `validatorama.getAccessToken()` with only 3 call sites. The one known limitation — stale session after logout on pages without a websocket (currently only the editor page).
- Since the server will be making more db calls during html rendering, we can take advantage of `res.on('finish')` to only perform necessary db writes after the response is sent already, e.g. updating the user's last active timestamp, or marking notifications as read.
- Unified way to make db backups. Have the server create a daily backup of the database, and right before every deploy. Store these backups in a `backups/` directory with timestamped filenames. Implement a retention policy to automatically delete backups older than 30 days to reduce disk usage.
- When you log out in another tab, all other tabs with a socket open, when they hear that logout alert, should, instead of swapping out the header content for the logged-out version, which would require hiding that stuff in the html document, instead, just do a full page reload via `window.location.reload()` so the server can re-render the page with the correct header and no need for client-side JS to hide/show elements.
